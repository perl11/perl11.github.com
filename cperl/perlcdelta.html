DESCRIPTION

For cperl also see Changes and perlcperl

None.

Core Enhancements from Perl5

See perldelta

Merged with perl-5.24.0, without the following perl-5.24.0 changes:

Remove legacy/dead code from B

This was explained on p5p. We need that old B code, it is not "dead". B modules are mainly on CPAN, and p5p is not able to maintain even B properly.

[perl #125341] check for unexpected trash after any sub start

We had a better fix for that, earlier.

remove-lexical-topic

We fixed the old lexical $_ bug (OPpTARGET_MY) already some months ago, in cperl5.22.2. It's harmful to remove it, as it is actually needed for given/when and smartmatch.

Remove unused filesystem stat symbols.

Reporting Bugs

If you find what you think is a bug, you might check the articles recently posted to the comp.lang.perl.misc newsgroup and the perl bug database at https://rt.perl.org/ . There may also be information at http://www.perl.org/ , the Perl Home Page.

If you believe you have an unreported bug, please run the perlbug program included with your release. Be sure to trim your bug down to a tiny but sufficient test case. Your bug report, along with the output of perl -V, will be sent off to perlbug@perl.org to be analysed by the Perl porting team.

If you think it's a cperl specific bug please file an issue at https://github.com/perl11/cperl/issues

If the bug you are reporting has security implications, which make it inappropriate to send to a publicly archived mailing list, then please send it to perl5-security-report@perl.org. This points to a closed subscription unarchived mailing list, which includes all the core committers, who will be able to help assess the impact of issues, figure out a resolution, and help co-ordinate the release of patches to mitigate or fix the problem across all platforms on which Perl is supported. Please only use this address for security issues in the Perl core, not for modules independently distributed on CPAN.

Looking at the security issues fixed in cperl only, the perl5-security-report@perl.org is only rarely able to asses and fix the issue you have, so can also email the cperl maintainer.
