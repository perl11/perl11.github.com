20 p2 is a she. She is a perl5 and possible perl6/nqp backend, based
21 on why the luck stiff's potion. See `README.potion` why potion is exciting.
22 Homepage: http://perl11.org/p2/

36 For most small examples comparable performance to lua and C.
37 For bigger examples optimizable via static types/early binding.
38 
39 Its exciting points for perl folks are:
40 
41  * Radically simplified and functional core.
42 
43  * Should parse most of perl5, just some XS and B tricks not.
44 
45  * Signatures, fast mop, types, native threads, call-replacing
46  macros for perl5.
47 
48  * No worries about pseudo-scientific discussions, project management,
49  performance bugs and feature deadlocks. Leaving the slack behind.
50 
51  * A lightweight generational GC, based on Basile Starynkevitch's
52  work on Qish, with ~4ms per GC on average with < 100MB heaps.
53  <http://starynkevitch.net/Basile/qishintro.html>
54  The compiler will add hooks for objects leaving scope and add
55  DESTROY method calls if they are visible at
56  compile-time. Run-time created DESTROY methods will not be
57  automatically called.
58 
59  * Using most of Damian Conway's p5i recommendations:
60  i.e. classes are immutable and final by default.
61  See <http://web.archive.org/web/20040207072905/http://www.yetanother.org/damian/Perl5+i/>
62 
63  * libp2 is usable as a fast backend for nqp/perl6, but parsing and supporting
64  p6 natively without going through nqp and the rakudo bootstrapping process
65  should be easier and faster.
66  Tokuhiro Matsuno already wrote a p6 parser, pvip.
67 
68  * Native and fast calling convention, no XS stack args on the heap.
69  XS is replaced by extern (dynamic ffi), and/or shared libraries
70  with normal function calls into the vm.
71 
72  * Just-in-time compilation for x86, x86-64 and ppc done. arm later.
73 
74  * Intermediate bytecode format and VM. Load and dump code. Decent
75  speed and cross-architecture. Heavily based on Lua's VM.
76 
77  * int, str, num are objects, you can optionally type all args and
78  lexical variables. In fact everything is an object as in parrot,
79  even the parsed AST, the vm state.
80 
81  * Sized arrays are non-autovivifying and initialized with undef,
82  resp. if typed 0, 0.0, "".
83 
84  my $a[3]; print $a[3] => compile-time error: array out of bounds
85  my int $i[3]; print $a[2] => 0
86  my num $n[3]; print $n[2] => 0.0
87  my str $s[3]; print $a[2] => ""
88 
89  * A new pragma "no magic" applies to all visible objects in scope.
90  {
91  no magic;
92  use Config;
93  print $Config{'ccflags'};
94  }
95  => compile-time error: Invalid use of tie with no magic
96 
97  use
98  {
99  no magic;
100  use Config ();
101  print $Config::Config{'ccflags'};
102  }
103  instead.
104 
105  * p6model: via scoped mixins, :multi, roles. Like Moose, just better
106  and roughly estimated 800x faster.
107  No multiple inheritance yet, \@ISA == 1.
108  Only one parent, depending on another parent, rather multiple
109  mixin compositions.
110 
111  * ~~ match works because all data are typed objects.
112  A destructuring-bind alike new match operator which matches
113  structures (list, trees, hashes of hashes) against structures,
114  values and types.
115 
116  * Bootstrapped "id" object model, based on Ian Piumarta's soda
117  languages. This means everything in the language, including
118  object allocation, parser, compiler and interpreter state are
119  part of the object model. (See COPYING for citations.)
120  In fact all objects and classes are closures, which simplifies
121  scoping, and makes p2/potion a functional vm.
122 
123  * parsers are scoped and run-time loadable for ffi, p6, and
124  other languages.
125 
126  * Interpreter is thread-safe and reentrant. This should
127  facilitate parallel interpreters initialized in a threadpool.
128 
129  * Small core. Under 10kloc. Install sloccount and run: `make sloc`.
130 
131  * Reified AST and bytecode structures. This is very important to
132  me. By giving access to the parser and compiler, it allows people
133  to target other platforms, write compilers, debuggers, code analysis
134  tools and even fully bootstrapped VMs. I'm not as concerned about
135  the Potion VM being fully bootstrapped, especially as it is tied
136  into the JIT so closely.
137 
138  * Memory-efficient classes. Stored like C structs. (Although the
139  method lookup table can be used like a hash for storing arbitrary
140  data.)
141 
142  * The JIT is also used to speed up some other bottlenecks. For
143  example, instance variable and method lookup tables are compiled
144  into machine code.
145 
146  * Fast asynchronous IO via libuv
147 
148 However, some warnings:
149 
150  * Ints cannot hold pointers and need to be boxed. This should be
151  transparent though. Use WeakRefs for pointers.
152 
153  * Strings are immutable (like Lua) and byte arrays are used for I/O
154  buffers. All strings and symbols are utf8 internally.
155 
156  * Limited platform support for coroutines.
157  This affects exceptions.
158 
159  * No regular expression bindings yet (pcre or sregex)
160 
161  * The parser is not GC safe yet. This affects eval.
162  Do not waste too much memory inside eval.
163 
164  * No OS threads yet.
165 
166  * No signal handling yet.
167 
168  * The compiler is mostly good enough, as in lua, but there are no
169  tradional compiler optimizations yet. We'd might need to add
170  constant folding and propagation, simple local basic block
171  optimizations, peephole optimizations, loop optimizations,
172  tracking side-effects. And tailcalls are not yet detected.
173 
174  * No type checks, bounds-check elimination and type optimizations
175  yet.
