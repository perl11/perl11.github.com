20 ## ~ potion ~
21 
22 Potion is an object- and mixin-oriented (traits)
23 language.
24 
25 Its exciting points are:
26 
27  * Just-in-time compilation to x86 and x86-64
28  machine code function pointers. This means
29  she's a speedy one. Who integrates very
30  well with C extensions.
31 
32  The JIT is turned on by default and is
33  considered the primary mode of operation.
34 
35  * Intermediate bytecode format and VM. Load
36  and dump code. Decent speed and cross-
37  architecture. Heavily based on Lua's VM.
38 
39  * A lightweight generational GC, based on
40  Basile Starynkevitch's work on Qish.
41  <http://starynkevitch.net/Basile/qishintro.html>
42 
43  * Bootstrapped "id" object model, based on
44  Ian Piumarta's soda languages. This means
45  everything in the language, including
46  object allocation and interpreter state
47  are part of the object model.
48  (See COPYING for citations.)
49 
50  * Interpreter is thread-safe and reentrant.
51  I hope this will facilitate coroutines,
52  parallel interpreters and sandboxing.
53 
54  * Small. Under 10kloc. Right now we're like
55  6,000 or something. Install sloccount
56  and run: make sloc.
57 
58  * Reified AST and bytecode structures. This
59  is very important to me. By giving access
60  to the parser and compiler, it allows people
61  to target other platforms, write code analysis
62  tools and even fully bootstrapped VMs. I'm
63  not as concerned about the Potion VM being
64  fully bootstrapped, especially as it is tied
65  into the JIT so closely.
66 
67  * Memory-efficient classes. Stored like C
68  structs. (Although the method lookup table
69  can be used like a hash for storing arbitrary
70  data.)
71 
72  * The JIT is also used to speed up some other
73  bottlenecks. For example, instance variable
74  and method lookup tables are compiled into
75  machine code.
76 
77 However, some warnings:
78 
79  * Strings are immutable (like Lua) and byte
80  arrays are used for I/O buffers.
81 
82  * Limited floating point support yet.
83 
84  * Limited error handling. I'm wary of just tossing
85  in exceptions and feeling rather uninspired
86  on the matter. Let's hear from you.

137 ## ~ items to understand ~
138 
139 1. A traditional object is a tuple of data
140  and methods: (D, M).
141 
142  D is kept in the object itself.
143  M is kept in classes.
144 
145 2. In Potion, objects are just D.
146 
147 3. Every object has an M.
148 
149 4. But M can be altered, swapped,
150  added to, removed from, whatever.
151 
152 5. Objects do not have classes.
153  The M is a mixin, a collection
154  of methods.
155 
156 Example: all strings have a "length"
157 method. This method comes with Potion.
158 It's in the String mixin.
159 
160 6. You can swap out mixins for the span
161  of a single source file.
162 
163 Example: you could give all strings a
164 "backwards" method. But just for the
165 code inside your test.pn script.
166 
167 7. You can re-mix for the span of a
168  single closure.
169 
170 To sum up:
171 
172 EVERYTHING IS AN OBJECT.
173 EVEN MIXINS ARE OBJECTS.
174 AND, OF COURSE, CLOSURES ARE OBJECTS.
175 
176 However, OBJECTS AREN'T EVERYTHING.
177 THEY ARE USELESS WITHOUT MIXINS.

180 ## ~ unique ideas (to be implemented) ~
181 
182 Potion does have a few unique features
183 underway.
184 
185 * It is two languages in one.
186 
187 The language itself is objects and closures.
188 
189  Number add = (x): self + x.
190 
191 But it also includes a data language.
192 
193  app = [window (width=200, height=400)
194  [button "OK", button "Cancel"]]
195 
196 The code and data languages can be interleaved
197 over and over again. In a way, I'm trying to find
198 a middle ground between s-expressions and stuff like
199 E4X. I like that s-expressions are a very light data
200 syntax, but I like that E4X clearly looks like data.
201 
202 When s-expressions appear in Lisp code, they look
203 like code. I think it is nice to distinguish the two.
204 
205 * Deeply nested blocks can be closed quickly.
206 I don't like significant whitespace, personally.
207 But I don't like end end end end.
208 
209  say = (phrase):
210  10 times (i):
211  20 times (j):
212  phrase print
213  _say
214 
215 The closing "_ say" ends the block saved to "say" var.
216 
217 Normally, blocks are closed with a period. In this case
218 we'd need three periods, which looks strange.
219 
220  say = ():
221  10 times:
222  20 times:
223  "Odelay!" print
224  ...
225 
226 If you prefer, you can give it some space. Or you can
227 use a variable name introduced by the block,
228 
229  say = (phrase):
230  10 times (i):
231  20 times (j):
232  phrase print
233  _ phrase
234 
235 
236  say = (phrase):
237  10 times (i):
238  20 times (j):
239  phrase print
240  _ i
241  .
242 
243 Maybe it all looks strange. I don't know. I'm just trying
244 things out, okay?
245 
246 * Elimination of line noise.
247 
248  I avoid @, #, $, %, {}.
249  Stick with ., |, (), [], =, !, ?. Easier on the eyes.
250  These are common punctuations in English.
251 
252 * I try to defer to English when it comes to punctuation rules.
253 
254 Period means "end". (In other langs it means "method call".)
255 Comma breaks up statements.
256 Space between messages gives a noun-verb feeling.
257 
258  window open (width=400, height=500)
259 
260 * Named block args
261 
262 
263  (1, 2, 3) map (item=x, index=i): i display, x + 1.
