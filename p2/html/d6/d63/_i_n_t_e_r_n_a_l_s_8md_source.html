2 # ~ the nature of potion's insides ~
3 
4 Potion's internals are based around the idea
5 that things that happen frequently should be
6 given the greater attention.
7 
8 For example, code execution happens more often
9 than code parsing and loading. Therefore, the
10 VM is larger and faster than the parser and
11 compiler.
12 
13 Other things that happen frequently:
14 
15  * Math.
16  * Conditionals.
17  * Method dispatch.
18  * Local variables.
19  * Strings, to build and compare.
20  * Memory allocation.
21 
22 More sporadic things likely include:
23 
24  * New methods.
25  * Creation of classes.
26  * Very deep scoping situations.
27  * Altering the inheritance tree.
28  * Evals.
29  * Code allocation.
30 
31 So understand that often the choices made in
32 Potion's VM are designed to accomodate the
33 first set rather than the second.

35 # ~
36 
37 This doc is going to touch quite a bit on
38 the JIT, compiling straight to x86 and PowerPC
39 instructions. So many languages are targetting
40 the JVM and .Net these days, or try to use 30MB
41 LLVM just for adding jit support. It made me
42 wonder what the hurdles are to generating machine
43 code directly.
44 
45 Clearly, the biggest benefit of having a common
46 runtime is garbage collection. The JVM and .Net
47 have some wild GC algorithms built in. And
48 that's actually a huge benefit. I only wish that
49 more garbage collectors were available as
50 libraries.
51 
52 Anyways, my favorite language has been Ruby
53 for many years, that's partially because
54 I like how easy it is to script from C,
55 for some efficiency where I need it. So
56 Potion is my little attempt to make a
57 language that feels easy, but compiles
58 to machine code, to give me the speed of
59 (un-optimized) C code and to use all the
60 C function call conventions so that I
61 can link to other C libs. I want to see
62 if its still possible to encroach on C.
63 
64 I should say that O'Caml and Haskell
65 already do a fine job as C competitors,
66 so it's not like anything here is that
67 new.

70 ## ~ lua's influence ~
71 
72 Potion uses a register-based bytecode VM that
73 is nearly a word-for-word copy of Lua's. The
74 code is all different, but the bytecode is
75 nearly identical. And Lua's 5.1 bytecode design is
76 heavily inspired by the 1997 Inferno OS "Dis VM".
77 <http://doc.cat-v.org/inferno/4th_edition/dis_VM_specification>
78 which was Lucent's faster and smaller competitor
79 of Sun's stack-based and GC-heavy JVM.
80 Dis/Limbo/Inferno concepts only got a recent boost
81 with Google adopting it as natively compiled Go.
82 
83 I also spelunked Neko VM for some ideas and
84 found this comparison very useful.
85 <http://nekovm.org/lua>
86 
87 I also took from the Lua the notion of using
88 tables for everything. And, like Lua, lists
89 and tables are interchangeable. But, in Potion,
90 I kept their representations separate.
91 
92 I also use immutable strings, like Lua.
93 
94 However, my object representation is more
95 like Neko or Ruby, where objects are all
96 represented by a word (32-bit on x86 and
97 64-bit on x86-64).

100 ## ~ the parts of potion ~
101 
102 1. the parser
103  - written in peg
104  (see `core/syntax.g`)
105  - produces a syntax tree
106  - the tree is a Potion object that can
107  be traversed in the code
108  - see `potion_source_load()`
109  in `core/compile.c`
110 
111 2. the compiler
112  - compiles a syntax tree into
113  bytecode (see `core/compile.c`
114  or `./potion -c -V code.pn`)
115  - has its own file format
116  (save to `.pnb` files)
117  - interception of keywords
118  happens here, rather than in
119  the parser, to simplify parsing
120 
121 3. the bytecode vm
122  - executes bytecode
123  - used mostly to ensure correctness
124  - helpful in debugging
125  - run code on non jitted vm with performance
126  penalty
127 
128 4. the jit vm
129  - compiles bytecode into a function
130  pointer
131  - uses the same argument passing strategy
132  as c functions
133 
134 5. the garbage collector
135  - generational, copying, near exact
136  - marks the stack, then uses a cheney
137  loop to catch everything else
138  - stack scanning is automatic (both
139  the jit and the bytecode vm keep
140  their registers on the call stack)
141  - liberal use of volatile for object
142  pointers, to allow them to move
143  after any allocation
144  - based on a collector called qish

147 ## ~ the jit ~
148 
149 potions jits intel x86 32+64-bit and ppc.
150 arm jit is in preparation.
151 ppc callcc/yield is buggy.
152 The X86 JIT is in `core/vm-x86.c`.
153 The jit translates Potion bytecode into
154 32-bit or 64-bit machine code.
155 
156 In the code:
157 
158  add = (x, y): x + y.
159 
160 The `add` closure is compiled into a c function
161 pointer looking like this:
162 
163  PN (*)(Potion *P, PN cl, PN self, PN x, PN y)
164 
165 This means you can actually load this closure
166 into C and call it directly.
167 
168  Potion *P = potion_create();
169  PN add = potion_eval(P, "(x, y): x + y.");
170  PN_F addfn = PN_CLOSURE_F(add);
171  PN num = addfn(P, add, 0, PN_NUM(3), PN_NUM(5));
172  printf("3 + 5 = %d\n", PN_INT(num));
173 
174 The macros are there to allow bytecode to be
175 wrapped in a function pointer, if needed. The
176 inside story looks like this:
177 
178  PN num = ((struct PNClosure *)add)->method(
179  P, add, 0, PN_NUM(3), PN_NUM(5));
180 
181 Note that calling from C into such methods directly
182 does not fill optional parameter values, it does not
183 check the parameter count and types.
184 If you need to use closures with optional parameters
185 and need to leave them out in the call, you need
186 to use the bytecode method.
187 
188 Wrong:
189 
190  PN add = potion_eval(P, "(x=N|y=N): x + y.");
191  PN_F addfn = PN_CLOSURE_F(add);
192  PN num = addfn(P, add, 0, PN_NUM(3), PN_NUM(5));
193  printf("3 + 5 = %d\n", PN_INT(num)); //ok
194  //PN num1 = addfn(P, add, 0, PN_NUM(3)); //wrong num1
195 
196 Better:
197 
198  long oldflags = (long)P->flags;
199  if (P->flags & EXEC_JIT) P->flags -= EXEC_JIT;
200  PN add = potion_eval(P, "(x=N|y=N): x + y.");
201  PN_F addfn = PN_CLOSURE_F(add);
202  PN num = addfn(P, add, 0, PN_NUM(3), PN_NUM(5));
203  printf("3 + 5 = %d\n", PN_INT(num)); //ok
204  PN num1 = addfn(P, add, 0, PN_NUM(3)); //ok
205  P->flags = (Potion_Flags)oldflags;

207 ## ~ the jit's assembly ~
208 
209 Now, let's talk about the compiled code.
210 
211 Earlier the example was,
212 
213  add = (x, y): x + y.
214 
215 So we have a function with two arguments
216 and a math operation.
217 
218 The C equivalent is,
219 
220  PN add(Potion *P, PN cl, PN self, PN x, PN y) {
221  PN reg1 = x;
222  PN reg2 = y;
223  reg1 = PN_NUM(PN_INT(reg1) + PN_INT(reg2));
224  return reg1;
225  }
226 
227 The arguments are moved to registers on the
228 stack. Then converted to ints. Then added.
229 And, lastly, the result is cast as a Potion
230 object.
231 
232 The machine code produced by the JIT will
233 be very close to the code produced by a
234 compiler with the optimizations turned off.
235 (_At least for the present this is true._)
236 
237 Like C functions, Potion keeps local variables
238 on the stack. The call stack is like a big tape
239 measure that goes on and on. Since this function
240 has two registers (reg1 and reg2,) we need
241 two inches from the tape measure.
242 
243 <http://en.wikipedia.org/wiki/Call_stack>
244 (See the parts that say "Local data storage"
245 and "Argument passing".)
246 
247 Our machine code says _"give me two word-sized slots in the stack"_
248 as the first order of business.
249 
250 We store `x` and `y` in those two slots. (See
251 the comment about C argument passing in the
252 next section.)
253 
254 Then we do the math. Even the `PN_INT` and
255 `PN_NUM` macros are fundamentally math.
256 
257 If you run potion with `-V` or `--verbose` on this script,
258 you'll see a bytecode disassembly.
259 
260  ; function ; 16 bytes
261  ; (x, y) 2 registers
262  .local "x" ; 0
263  .local "y" ; 1
264  [1] getlocal 0 0
265  [2] getlocal 1 1
266  [3] add 0 1
267  [4] return 0
268 
269 Each of these lines corresponds to a line of
270 C code inside that function a few paragraphs
271 back. You see how they line up there?

274 ## ~ x86 and x86-64 ~
275 
276 64-bit CPU instructions are a superset of
277 32-bit CPU instructions. And many 32-bit
278 instructions can be converted to 64-bit by
279 prefixing them.
280 
281 Take this code,
282 
283  8b 55 f8 // mov %ebp[-1] %edx
284  8b 45 fc // mov %ebp[-2] %eax
285  83 e8 04 // sub 0x4 %edx
286  89 50 04 // mov %edx %eax[1]
287 
288 These are the 32-bit instructions for altering
289 an upval. (_A variable in upper scope_.) Since
290 a closure doesn't have access to the registers
291 of another function, these variables are passed
292 as pointers (the `PNWeakRef` struct.)
293 
294 This code loads the upval into `%eax` and the new
295 value into `%edx`. To get the upval's actual struct
296 pointer, we subtract 4 from `%edx`. Finally, we
297 move the new value into offset 4 in the struct
298 pointed to by the upval.
299 
300 In 64-bit code this would be,
301 
302  48 8b 55 f0 // mov %rbp[-1] %rdx
303  48 8b 45 f8 // mov %rbp[-2] %rax
304  83 e8 04 // sub 0x4 %edx
305  48 89 50 08 // mov %rdx %rax[1]
306 
307 All it took was adding the `0x48` prefix to our
308 mov instructions and changing our offsets to
309 measure 8 bytes rather than 4. The third
310 instruction actually stays the same, since
311 we're only doing math on the lower 32-bits of
312 the upval.
313 
314 We now call the registers `%rax` and `%rdx`.
315 That's just what the CPU folks call `em`.
316 
317 The only other technique that changes between
318 32-bit and 64-bit is function calling. On 32-
319 bit, we keep all the function arguments on
320 the stack. On 64-bit, some function arguments
321 are passed through registers.
322 
323 That's the purpose behind the function named
324 `potion_x86_c_arg`. To shuttle the arguments
325 in and out, depending on the architecture.

350 ## ~ the garbage collector ~
351 
352 Potion's garbage collector is based on the
353 work Basile Starynkevitch performed with his
354 small generational GC entitled *Qish*. As with
355 everything else in Potion, the goal isn't to
356 be the best, but to experiment with what sort
357 of advanced features can be squeezed into a
358 language while keeping the code compact.
359 
360 Since Potion interpreters are per-thread,
361 each interpreter has its own GC heap. This
362 allows interpreters to run in parallel and
363 to start up and shut down independently.
364 Perhaps one day the GC will run in parallel.
365 
366 Qish is a two-finger generational collector.
367 (Also called a _Cheney loop_.) This collector
368 only has two generations. When the time for
369 garbage collection arrives, everything from
370 the call stack is scanned and moved from the
371 young gen to the older gen. The insides of
372 all those objects are then scanned and moved
373 as well. The goal is simply to wipe out the
374 birth region and replace it with another.
375 
376 What about objects from the birth region
377 that are stored off in some old hashtable
378 somewhere? Or stuff added to a global object?
379 
380 The birth region has a write barrier: an
381 array of pointers to any object which has
382 recently changed. If a new object is added
383 to a tuple, it's the tuple's job to notify
384 GC so that the tuple will get scanned on
385 the next GC pass.
386 
387 What happens when the old generation runs
388 out of room? A new old generation is
389 allocated and we do a major collection:
390 the call stack is scanned and everything from
391 the birth region AND the full-to-the-brim
392 older region are both scanned and moved on
393 to the new generation.
394 
395 This all has some obvious pitfalls. First,
396 that's a lot of memory moving around all
397 the time -- pointers moving around. And,
398 second, that's a lot of scanning to do:
399 does the entire heap have to be scanned
400 every time there's a major collection?
401 
402 Yes, it's a lot of work and I hope to get
403 some real benchmarks and determine a new
404 strategy to buy some speed: either through
405 introducing a third generation or by
406 defragmenting memory or by allowing pages
407 to be partially unmapped. Probably the last
408 one: I'll weight sections of the older
409 generation and then clear out those sections
410 as they become more sparse.
411 
412 As for moving pointers around, that's
413 why the volatile keyword is a part of the
414 PN typedef.
415 
416 Does Potion perform *exact GC*? Not strictly.
417 During the stack scanning phase, Potion has
418 no idea what pointers are legit. For now,
419 this is resolved by investigating the header
420 of each pointer's object for a valid type
421 object. The type can then be compared with
422 the contents of the object to be sure the
423 object is real. To me, this feels exact
424 enough. (If, in practice, it proves to
425 be too messy, I may resort to using a
426 macro to identify exact pointers, just as
427 Qish does.)

430 ## ~ the volatile keyword ~
431 
432 Since objects move freely about, Potion
433 requires use of the *volatile* keyword for
434 all access to object ids (`PN`) from C.
435 As most people don't readily understand
436 the volatile keyword, let's go over it.
437 
438 There are three common uses of the volatile
439 keyword in Potion code:
440 
441  PN string = potion_byte_str("...");
442 
443 Since `volatile` is part of the PN typedef
444 this code actually means:
445 
446  volatile unsigned long string =
447  potion_byte_str("...");
448 
449 In this case, the `volatile` keyword will
450 generally prevent the PN from being stored
451 in a system register when the code is
452 optimized. Even for caching purposes.
453 
454 So any time the `string` variable is used,
455 it'll be looked up directly on the stack.
456 This means the code won't be optimized fully,
457 but it also means the value will be looked
458 up every time we use it, in case GC has changed
459 the object's place on us.
460 
461  struct PNObject * volatile string = ...;
462 
463 If we cast the string to a `PNObject` pointer,
464 we need to be sure to place the `volatile`
465 keyword after the star. This tells the compiler
466 that the _pointer_ is the volatile one, not the
467 struct itself.
468 
469  struct PNUserType {
470  PN_OBJECT_HEADER
471  PN a, b, c;
472  };
473 
474 In any structs which are wrapped by
475 a Potion object, you need to be sure that any
476 object handles are marked volatile as well.
477 In the above case, the PN typedef handles that.
478 
479 Also be sure to not nest function calls which return PN
480 objects. With an optimizing (gcc/clang -O3) such
481 intermediate results might be stored in a register, as
482 volatile only applies to lvalues and the GC will not see this
483 PN and move references to it away.
484 Use temp. PN variables instead.
485 
486 This is a bit tedious to keep in mind, but
487 is compensated for by Potion's automatic stack
488 scanning (a technique which has brought easy
489 C extensions to Ruby) and object convention.
490 If your object pointers are all 8-byte aligned,
491 you won't need to write a GC marking function.
492 
493 ## ~ the fixed memory section ~
494 
495 Potion does have one other section of
496 memory: a permanent generation containing
497 all of the objects that are used to bootstrap
498 Potion's environment. The Lobby object, the
499 class lookup table, the GC structures, etc.
500 
501 This section of memory is positioned at the
502 beginning of the first birth area. After the
503 birth area, fills that memory is detached
504 (like the saucer section of the Enterprise)
505 and lives its life independantly of the other
506 regions, not to be moved or released until
507 the Potion interpreter is destroyed.

510 ## ~ object overhead ~
511 
512 Primitives (such as numbers, booleans and nil) are
513 immediate values in Potion, kept in a single-word stack
514 slot.
515 
516 Everything else is given a spot in memory. Every
517 allocated object has a single word header specifying
518 its type. Here's a breakdown of what each object looks
519 like on 32-bit:
520 
521  PNObject = 4 + (N * 4) bytes
522  4 bytes (type = 0x250009)
523  N * 4 bytes (each object field)
524 
525  PNWeakRef = 8 bytes
526  4 bytes (type = 0x250004)
527  4 bytes (pointer to object)
528 
529  PNString = 13 + N bytes
530  4 bytes (type = 0x250003)
531  4 bytes (length of string)
532  4 bytes (an incremental id used as hash)
533  N + 1 bytes (UTF-8 + '\0')
534 
535  PNBytes = 9 + N bytes
536  4 bytes (type = 0x25000c)
537  4 bytes (length of string)
538  N + 1 bytes (string + '\0')
539 
540  PNTuple = 8 + (N * 4) bytes
541  4 bytes (type = 0x250006)
542  4 bytes (length of tuple)
543  N * 4 bytes (each tuple item)
544 
545  PNClosure = 16 + (N * 4) bytes
546  4 bytes (type = 0x25005)
547  4 bytes (function pointer)
548  4 bytes (signature pointer)
549  4 bytes (length of attached data)
550  N * 4 bytes (attached data)
551 
552  PNData = 8 + N bytes
553  4 bytes (type = 0x250011)
554  4 bytes (length of contained data)
555  N bytes (data)
556 
557 The smallest allocation unit in Potion is
558 16 bytes. Everything larger than 16 bytes
559 is aligned to 8 bytes. This is true for both
560 32-bit and 64-bit platforms.
561 
562 The reason for this figure is to allow room for
563 the `PNFwd` struct:
564 
565  unsigned int // POTION_FWD or POTION_COPIED
566  unsigned int // allocated space in this slot
567  unsigned long // pointer to moved object
568 
569 Any time an object is moved by GC or reallocated,
570 a temporary `PNFwd` is placed in the memory slot to
571 redirect callers temporarily. In practice, you
572 shouldn't ever need to deal with these values,
573 since the stack scanning phase will rewrite them
574 automatically.
575 
576 The first type of `PNFwd` is `POTION_FWD`. This type
577 is used for reallocation. Potion only has four
578 built-in types that are subject to reallocation:
579 tuples, tables, byte strings, and internal buffers.
580 (Internal buffers are used by the JIT to store
581 machine code output and by the object model to
582 keep an array of type classes in circulation.)
583 
584 The `POTION_FWD` pointers will happen between GC
585 cycles, so they stay in place for awhile. The
586 API around those four built-ins is designed to
587 check for forwarded pointers whenever they're
588 passed in as an object. One the next GC cycle
589 hits all the pointers will be updated to use
590 the new objects (even if `POTION_FWD` occurs in
591 the old generation, due to the write barrier.)
592 
593 The other `PNFwd` type is a `POTION_COPIED` pointer.
594 This is used by the GC to denote an object which
595 has moved to the old generation. These should
596 not appear outside the GC cycle. The `PNFwd` is
597 placed immediately when the object is moved and
598 is then used to rewrite pointers found on the
599 stack or inside other objects. After the GC
600 cycle, these pointers will disappear when the
601 its outdated generation is discarded.
